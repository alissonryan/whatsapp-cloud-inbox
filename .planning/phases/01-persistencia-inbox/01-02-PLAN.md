  <name>Task 1: Criar verificador de assinatura X-Hub-Signature-256</name>
  <files>
src/lib/webhooks/meta-signature.ts
  </files>
  <action>
Implementar um helper `verifyMetaSignature({ appSecret, rawBody, header })` que:

- Leia o header `X-Hub-Signature-256` no formato `sha256=...`.
- Calcule `HMAC-SHA256(rawBody, appSecret)` e compare usando timing-safe equal.
- Retorne boolean e/ou lance erro com mensagem clara.

Obs: usar o raw body (bytes) sem JSON parse antes do HMAC.
  </action>
  <verify>
Criar um teste rapido via script ou chamada local (pode ser manual) para validar que a funcao retorna true/false com header correto/incorreto.
  </verify>
  <done>
Assinatura pode ser validada com seguranca (timing-safe) e usando raw body.
  </done>
</task>

<task type="auto">
  <name>Task 2: Implementar rota /api/webhooks/meta (GET verify + POST signature check)</name>
  <files>
src/app/api/webhooks/meta/route.ts
  </files>
  <action>
Criar `route.ts` com:

- GET:
  - Ler `hub.mode`, `hub.verify_token`, `hub.challenge`
  - Se token bater com `META_WEBHOOK_VERIFY_TOKEN`, responder com o challenge (200)
  - Caso contrario, responder 403

- POST:
  - Ler raw body via `request.arrayBuffer()` (antes de parsear JSON)
  - Validar assinatura `X-Hub-Signature-256` com `META_APP_SECRET`
  - Parsear JSON (apos validacao)
  - Chamar um handler de ingestao (Task 3)
  - Responder 200 rapido (mesmo que o handler faca upserts)
  </action>
  <verify>
`npm run build` e `curl` local (GET verify) retorna o challenge quando o token bate.
  </verify>
  <done>
Endpoint suporta verificacao e rejeita requests sem assinatura valida.
  </done>
</task>

<task type="auto">
  <name>Task 3: Ingestao idempotente de messages/statuses no DB</name>
  <files>
src/lib/webhooks/meta-whatsapp.ts
  </files>
  <action>
Implementar `ingestMetaWhatsAppWebhook(payload)` que:

- Navegue no payload ate `entry[].changes[].value`.
- Suporte:
  - `messages[]` (inbound)
  - `statuses[]` (sent/delivered/read/failed/played)
- Para cada message/status:
  - Resolver `tenantId` via `getOrCreateDefaultTenant()`
  - Upsert de `Contact` (wa_id, nome quando existir)
  - Upsert de `Conversation` (tenant + phone_number_id + wa_id) e atualizar `lastActiveAt`
  - Upsert de `Message` por `metaMessageId` (dedup)
  - Inserir/atualizar `MessageStatus` (dedup por (messageId + status + timestamp))
- Persistir dados suficientes para:
  - listar conversas com preview (last message)
  - listar mensagens por conversa (ordenadas por timestamp)
  </action>
  <verify>
Simular payloads (fixtures) e garantir que rodar duas vezes nao duplica registros (idempotencia).
  </verify>
  <done>
Webhook alimenta o DB e suporta retries sem duplicar mensagens/statuses.
  </done>
</task>
